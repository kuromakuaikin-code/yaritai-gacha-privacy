# 盆踊り・地域祭りナビ東海 — 運営マニュアル

サイトの目的：**「東海4県で、盆踊りと地域に根付く祭りを大小問わず見つけられること」**
対象地域は愛知県・岐阜県・三重県・静岡県。盆踊り、神社・寺院の祭礼、山車祭、七夕祭、地域の季節行事を扱う。`category` は `bonodori` または `festival` を指定する。

## 構成

```
bon-odori/
├── index.html      一覧ページ（今日／今週末／市町村／近い順）
├── submit.html     投稿案内ページ（GoogleフォームURLを設定して使う）
├── events.json     ★確認できたイベントのデータ本体
├── coverage.json   ★地域ごとの公式情報確認状況
├── candidates.json ★未確認の盆踊り候補
├── build.py        個別ページ生成スクリプト
├── e/              生成された個別イベントページ（直接編集しない）
├── sitemap.xml     生成される（直接編集しない）
├── drafts/         掲載前の候補メモ（サイトには表示されない）
└── ADMIN.md        このファイル
```

## 日常の運用（イベントの追加・修正・非公開）

確認できた催しは `events.json`、まだ確認できていない地域は `coverage.json` を更新する。

- 編集はスマホ・PCの GitHub Web 画面からでもOK（リポジトリで events.json を開く → ✏️ → Commit changes）
- コミットすると GitHub Actions（`.github/workflows/build-bon-odori.yml`）が個別ページと sitemap を**自動で再生成**する
- ローカルで確認したい場合のみ `cd bon-odori && python3 build.py` を手動実行

- **追加**: 配列に1件足す。`id` は英数字とハイフン（例 `nagoya-midori-narumi-gakku`）。URLになるので後から変えない
- **修正**: 該当項目を直し、`lastVerified` を今日の日付に更新
- **非公開**: `status` を `"hidden"` にする（データは残る。build.py が個別ページを自動削除する）
- **中止・延期**: 消さずに `eventStatus` を `"cancelled"` / `"postponed"` にして `statusNote` に理由を書く（中止情報こそ利用者に価値がある）
- **承認待ち**: 投稿されたがまだ確認できていないものは `status: "pending"` で入れておく（非公開のまま管理できる）

### 未確認地域の更新

- `coverage.json` は愛知県54市町村を網羅し、名古屋市は16区に分けている
- `unknown`: 2026年の公式情報を確認できていない
- `waiting`: 開催予定や講習会は確認できたが、本番の日時・会場などの公式発表待ち
- `review`: 情報の矛盾や対象範囲を確認中
- 同じ市・区で `events.json` の公開イベントが1件以上ある場合、画面では自動的に未確認地域から外れる
- 「公式情報が存在しない」とは断定せず、「現時点で確認できていない」と表示する

### 未確認候補の更新

- イベント名や過年度の公式ページは見つかったが、2026年の日時・会場・内容が確定していない候補は `candidates.json` に入れる
- `reason` に未確認の理由を明記し、確定イベントと誤認されないようにする
- 投稿から確認待ちの `events.json` の `status: "pending"` も、未確認候補として同じタブに自動表示される
- 公式確認できたら `events.json` に `published` で追加し、`candidates.json` から候補を削除する

## 毎年のデータ流用

前年のデータは消さない。翌年は同じエントリの以下だけ更新する：

1. `dates` を新しい日程に
2. `lastVerified` を確認日に
3. 必要なら `eventStatus` を `scheduled` に戻す

過去日程のイベントは一覧に自動で表示されなくなるだけなので、シーズン外もそのままでよい。
`id` を変えないことで個別ページのURLが年をまたいで維持され、Google検索の評価も引き継がれる。

## 投稿の受付（Googleフォーム）

1. Googleフォームを作成。質問は submit.html の5項目に合わせる：
   - チラシ画像（ファイルアップロード・任意）
   - イベント名（必須）
   - 日時（必須・記述式）
   - 場所（必須・記述式）
   - 情報元（必須・記述式：回覧板／掲示板／主催者本人／URLなど）
   - 任意：あなたの立場（主催者／住民）、画像の掲載許可（主催者本人 or 許可を得ている）
2. フォームURLを `submit.html` 内の `const FORM_URL = ""` に貼る
3. 回答はスプレッドシートに溜まる → 下の「掲載判断」を経て events.json に手で転記する

## 掲載判断のルール（重要）

- **必ず一次情報（主催者・自治体・町内会・神社・施設の公式発表）を探して確認**してから掲載する
- 自治体・公式観光協会が主催者情報を確認して掲載しているページも公式情報として扱う
- 公式ページを情報元にする → `posterType: "admin"`（画面表示は「公式情報あり」。開催保証の意味ではない）
- 主催者本人から提供された情報 → `"organizer"`（主催者からの情報）
- 公式情報が見つからない → `posterType: "resident"`（住民からの情報・未確認）として区別して掲載するか、`status: "pending"` で保留
- 競合サイト・まとめサイトで見つけた場合も、**そこからは転載しない**。入口としてだけ使い、公式情報を探し直す
- `desc`（紹介文）は公式発表・投稿・自分の確認に基づき**自分の言葉で**書く。他サイトの紹介文のコピー・言い換えは禁止
- 画像は投稿者が権利を持つか掲載許可が確認できた場合のみ `imageAllowed: true` にして掲載

### 禁止事項

- 競合サイトからの自動取得・一括コピー・スクレイピング
- 他サイトの紹介文の言い換え転載、独自のまとめ方の流用
- チラシ画像・写真・ロゴの無断掲載
- Googleマップ等の画面キャプチャ掲載（リンクはOK）
- 情報元を隠した掲載
- 公式ページのキャッチコピーや特徴的な文章を、語尾だけ変えて掲載すること

### 公式情報から安全に要約する手順

1. 開催名・日付・時刻・会場・雨天対応などの事実だけをメモする
2. 元ページを閉じても書ける程度の短い文章を、自分の構成で作る
3. 公式ページ固有のキャッチコピー、長い歴史説明、見どころ紹介は使わない
4. `source.url` に確認した公式ページを直接指定し、`lastVerified` を更新する
5. 画像は使わず `imageAllowed: false` を基本とする。書面等で許可を得た場合だけ掲載する

## 仮データについて

`sample: true` のイベントがある場合だけ、一覧上部に「仮データ掲載中」の注意書きが表示される。公開前に仮データを削除するか `status: "hidden"` にする。
